# CLI Authentication for Personal Servers

**Date:** 2026-03-21
**Status:** Design / RFC

## Problem

The CLI (`vana`) needs to authenticate with Personal Servers in three modes:

1. **Local PS** (localhost) — currently works with no auth
2. **Cloud PS** (myvana.app, managed by account.vana.org) — user's wallet is Privy-managed, no private key available
3. **Self-hosted PS** (user's own server) — user may or may not have wallet access

The cloud PS is the critical gap. The PS requires `Authorization: Web3Signed` headers for owner-only operations (data writes, grant management). The CLI can't sign these because the Privy embedded wallet's private key is never exposed.

## Industry Research

### Cloud-first CLI tools

Every best-in-class CLI uses **OAuth 2.0 Device Code Flow (RFC 8628)** for browser-based login:

| Tool | Protocol | Token Return | Localhost Required? |
|------|----------|-------------|-------------------|
| GitHub CLI | Device Code | API polling | No |
| Vercel CLI | Device Code | API polling | No |
| Codex CLI | Device Code (+ browser) | Polling / callback | Optional |
| Claude Code | OAuth PKCE | Redirect | Needs browser |
| Supabase CLI | Custom ECDH polling | Encrypted polling | No |

### Cloud + self-hosted dual-mode tools

Tools that support both cloud-hosted and self-hosted instances:

| Tool | Cloud auth | Self-hosted auth | Instance selection |
|------|-----------|-----------------|-------------------|
| GitLab (glab) | Device code → gitlab.com | Same flow → self-hosted instance | `--hostname` flag |
| Bitwarden CLI | `bw login` → bitwarden.com | `bw config server <url>` then `bw login` | `bw config server` |
| Docker | `docker login` → hub.docker.com | `docker login registry.example.com` | URL as argument |
| Tailscale | `tailscale up` → tailscale.com | `tailscale up --login-server <url>` | `--login-server` flag |
| MinIO | `mc alias set cloud ...` | `mc alias set myserver ...` | Named aliases |
| Nextcloud | N/A (self-hosted only) | Login Flow v2: POST /login/v2, browser opens, CLI polls | Server URL in client |

**Key patterns:**
1. **Same flow, different target:** GitLab, Docker, Tailscale use the same auth flow for both cloud and self-hosted — just pointed at a different URL
2. **Named profiles:** MinIO's alias system is the best multi-instance UX — named references with independent credentials
3. **Server-initiated auth:** Nextcloud's Login Flow v2 is the gold standard for self-hosted — the server provides the auth flow, the CLI just follows it

### The Nextcloud Login Flow v2 pattern (most relevant for self-hosted)

1. CLI POSTs to `https://my-server.example.com/login/v2`
2. Server returns a `login` URL and a `poll` endpoint + token
3. Browser opens the login URL — user authenticates (supports SSO, MFA, whatever the server has)
4. CLI polls the poll endpoint until auth completes
5. Server returns an app-specific credential
6. CLI stores it locally

This is RFC 8628 but where **the server itself is the auth provider**. The CLI doesn't need to know what auth method the server uses — it just opens a browser and polls.

**Implications for Vana:**
- Cloud PS auth goes through account.vana.org (device code flow)
- Self-hosted PS auth goes through the PS itself (Nextcloud-style login flow)
- The CLI detects which flow to use based on the server URL
- Same UX for the user in both cases: code + browser + poll

**The pattern:**
1. CLI displays a short code and URL
2. User visits URL in any browser (any device — works in containers, SSH, WSL)
3. User authenticates and enters code
4. CLI polls API until token is granted
5. Token stored locally (OS keyring with plaintext fallback)
6. Every tool also provides a `*_TOKEN` env var for CI/automation

## Design

### Architecture

```
vana login
   |
   v
CLI generates device code → POST /api/auth/device
   |
   v
Browser opens: account.vana.org/auth/device?code=XXXX
   |
   v
User logs in via Privy, enters code, approves
   |
   v
CLI polls: GET /api/auth/device/poll?device_code=...
   |
   v
account.vana.org returns:
  - session_token (for future API calls)
  - personal_server_url (user's cloud PS, if provisioned)
  - ps_access_token (fresh 30-day CLI session token for that PS)
  - expires_at
   |
   v
CLI stores credentials in ~/.vana/auth.json
   |
   v
CLI uses ps_access_token for PS requests:
  Authorization: Bearer <ps_access_token>
```

### PS Access Token

Rather than proxying every request through account.vana.org for signing, we issue a **fresh PS CLI session token** at login time:

- Generated by account.vana.org on each successful `vana login`
- Provisioned into the running PS token store through a separate control-plane token
- The PS trusts the CLI session token as equivalent to owner auth
- The CLI gets the token via the device code flow
- Token is valid for 30 days, then the CLI must re-login
- Token is scoped to the specific PS instance

This avoids:
- Signing proxy latency on every CLI request
- account.vana.org being a runtime dependency for CLI → PS communication
- Complex Web3Signed header construction in the CLI

### Three Modes

**1. Local PS (localhost)**
- No auth (unchanged)
- Auto-detected via port scan
- Works without `vana login`

**2. Cloud PS (myvana.app)**
- Requires `vana login` first
- Device code flow against account.vana.org → gets ps_access_token
- CLI auto-discovers PS URL from account.vana.org
- Uses `Authorization: Bearer <ps_access_token>` for all PS requests
- If token expires, CLI prompts re-login

**3. Self-hosted PS**
- User sets URL via `vana server set-url <url>`
- CLI detects it's not a myvana.app URL → initiates auth against the PS itself
- Same UX as cloud: `vana login` opens browser, user authenticates, CLI gets token
- The PS has its own `/login/v2` endpoint (Nextcloud Login Flow v2 pattern):
  1. CLI POSTs to `https://my-ps.example.com/login/v2`
  2. PS returns `{ login_url, poll_endpoint, poll_token }`
  3. Browser opens `login_url` — user signs with their wallet (or whatever auth the PS supports)
  4. CLI polls `poll_endpoint` until auth completes
  5. PS returns `{ access_token }` — a Bearer token scoped to this PS
  6. CLI stores the token in `~/.vana/auth.json`
- Also supports `VANA_PS_TOKEN` env var for manual token management

### API Routes (account.vana.org)

```
POST /api/auth/device
  → { device_code, user_code, verification_uri, expires_in, interval }

GET  /api/auth/device/poll?device_code=...
  → { status: "pending" }
  → { status: "authorized", session_token, personal_server_url, ps_access_token, expires_at }
  → { status: "expired" }
```

### PS Access Token Flow

**At provision time:**
1. account.vana.org generates a random control-plane token (stored today in `personal_servers.access_token`)
2. That control-plane token is passed to the VM as instance metadata
3. PS reads the token from metadata / `PS_ACCESS_TOKEN`
4. account.vana.org uses that control-plane token to provision future CLI session tokens into the running PS

**At login time:**
1. CLI completes device code flow
2. account.vana.org generates a fresh CLI session token for that PS
3. account.vana.org calls `POST /auth/device/token` on the PS using the control-plane token
4. PS stores the CLI session token with a 30-day expiry
5. account.vana.org returns that fresh CLI session token to the CLI
6. CLI stores it in `~/.vana/auth.json`

**At request time:**
1. CLI sends `Authorization: Bearer <ps_access_token>` to the PS
2. PS validates the token against its stored copy
3. If valid, request is treated as owner-authenticated
4. `vana logout` revokes that CLI session token via `DELETE /auth/device/token`

### Credential Storage

```json
// ~/.vana/auth.json
{
  "account": {
    "address": "0x2Ab394e4Be7C43AC360D226a31E1c90BC01aafa1",
    "session_token": "vana_sess_...",
    "expires_at": "2026-04-20T..."
  },
  "personal_server": {
    "url": "https://0x2ab3...fa1.myvana.app",
    "access_token": "vana_ps_...",
    "expires_at": "2026-04-20T..."
  }
}
```

File permissions: `0600` (owner read/write only).

### Env Var Escape Hatch

For CI/automation:
- `VANA_SESSION_TOKEN` — bypasses browser login
- `VANA_PS_TOKEN` — direct PS access token (skips account.vana.org entirely)
- `VANA_PS_URL` — PS URL override (existing `VANA_PERSONAL_SERVER_URL`)

### Security Considerations

1. **Device code brute force:** Use 8-char alphanumeric codes with 5-minute expiry and rate limiting
2. **PS access token scope:** Token is scoped to one PS instance. Revoking the token (via account.vana.org UI or `vana logout --revoke`) invalidates it immediately
3. **Per-login session tokens:** On `vana login`, account.vana.org issues a fresh CLI session token. Multiple concurrent CLI sessions are allowed; each token is independently revocable and expires after 30 days.
4. **Local storage:** File permissions `0600`. OS keyring preferred if available (future enhancement)
5. **Transport:** All communication over HTTPS. PS access tokens are never logged or displayed in full

### Explicit Non-Goals

- Token refresh. The launch contract is explicit 30-day expiry with re-login after expiration.
- OS keyring integration (launch uses file-based storage with `0600` permissions)
- MFA/2FA beyond the underlying identity providers (Privy for cloud, wallet auth for self-hosted)
- Named profiles / multi-instance alias management
